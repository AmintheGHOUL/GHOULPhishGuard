Real phishing detection test set

Create a controlled dataset with these categories.

Group A — obvious phishing

These should score high risk.

Test A1
From: Microsoft Security <security@microsoft.com>
Reply-To: microsoft-support@account-security-check.com
Body says account will be closed in 24 hours
Link: https://login.microsoftonline.com.security-check-login.com/reset

Expected:

flagged as phishing

reasons mention fake domain, urgency, reply-to mismatch

Test A2
From: PayPal <security@paypal.com>
Link text says paypal.com
Actual href is https://paypal-login-safe-check.com

Expected:

flagged as phishing

reason explains visible link mismatch

Test A3
Attachment named Invoice_2026.zip

Expected:

risky attachment warning

not necessarily “phishing” by attachment alone, but clearly suspicious

Group B — harder phishing

These test whether your logic is actually smart.

Test B1
From: docusign@docusign.com
Reply-To: noreply@docusign.com
Link: https://docusign.secure-document-view.com/?session=992821

Expected:

flagged because real domain is secure-document-view.com

session parameter alone should not be the main reason

Test B2
From: security@company.com
Body is well-written and professional
Link goes to a Microsoft form or Google Sites page used as a redirect

Expected:

at least suspicious

should mention abuse of legitimate platform if implemented

Test B3
Email says:
“Hi Amin, we reviewed your recent application and need you to confirm payroll details.”
No bad grammar. No obvious spoofing.

Expected:

content-analysis engine should catch unsolicited sensitive request

should not rely only on grammar mistakes

Group C — legitimate emails that must not be over-flagged

These matter a lot.

Test C1
A real Gmail password alert from Google

Expected:

low risk or at most mild caution

should not scream phishing just because of urgency wording

Test C2
A real DocuSign email

Expected:

safe or low risk

should parse legitimate domains properly

Test C3
A marketing email sent through SendGrid or Mailchimp

Expected:

should not mark phishing only because Return-Path differs from From

This one is critical because many beginners get this wrong.
